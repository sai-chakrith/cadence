BRD requirements not yet in your plan — fix these first

These are all explicitly listed in the BRD. Evaluators will check them one by one under Parameter 2. Each missed rule is a direct score drop.
G1

Weightage validation — three hard rules

High risk

Total weightage across all goals must equal exactly 100%. Each goal minimum 10%. Maximum 8 goals per employee. Enforce at both frontend (live counter) and backend (reject API call if violated). Show clear inline error messages — evaluators will deliberately try to break this.

G2

UoM progress score formulas

High risk

Four formula types must be implemented: Min (Achievement ÷ Target), Max (Target ÷ Achievement), Timeline (completion date vs deadline), Zero (0 achievement = 100%, else 0%). These drive the computed progress scores shown in check-ins. Hard-code the logic server-side and display it clearly.

G3

Shared goals — distinct data model

Must-have

Admin or Manager pushes a departmental KPI to multiple employees. Each recipient gets a copy where only weightage is editable — Title and Target are read-only. When the primary owner logs achievement, it syncs to all subscriber copies. Needs a goal_assignments join table with owner_id and subscriber_id fields.

G4

Check-in window enforcement

Must-have

The portal must only allow achievement updates during the defined windows: July (Q1), October (Q2), January (Q3), March/April (Q4/Annual). Outside these windows the form should be read-only with a "window closed" message. Configurable via the cycle config table.

G5

Manager structured check-in comment

Must-have

After reviewing Planned vs. Actual, the Manager must be able to add a structured Check-in Comment to document the discussion. This is separate from the approval workflow — it's a logged comment per quarterly check-in, stored with cycle reference and timestamp.

G6

Demo accounts — one complete journey per role

Demo risk

The submission requires login credentials for all 3 roles with a complete user journey. Seed the DB so evaluators can log in as Employee (goals created + submitted), Manager (goals approved + check-in comment added), and Admin (cycle configured + audit trail visible) without any setup.

How to frame the RAG layer to maximise hackathon scores

The BRD explicitly asks for Analytics (§5.4) and an Audit Trail (§4). Your RAG layer covers both — and extends them. This is the story to tell evaluators.

5 killer queries to demo

1

"Show me the Q2 completion dashboard for the Sales department"

→ Returns a live QoQ bar chart inline. Covers BRD §4 Completion Dashboard.

2

"Which employees have not submitted goals within 5 days of the cycle opening?"

→ Returns a list with days overdue. Covers BRD §5.3 Escalation Module trigger.

3

"What changed in Priya's Q2 goal sheet after the lock date?"

→ Returns the audit trail narrative for that goal. Covers BRD §4 Audit Trail.

4

"Compare check-in completion rates across all L1 managers this quarter"

→ Returns a ranked table. Covers BRD §5.4 Manager Effectiveness Dashboard.

5

"Show goal distribution by Thrust Area for Engineering — Q1 vs Q2"

→ Returns a grouped chart. Covers BRD §5.4 Goal Distribution Analysis.

What to index

A

Goal summary chunks

One prose chunk per employee per cycle — name, dept, all goals with status and completion %.

B

Audit event windows

All post-lock changes grouped by goal_id as a chronological narrative.

C

Dept × cycle rollups

Pre-aggregated completion rate, submission rate, overdue count — refreshed daily.

Minimal viable RAG stack

1

pgvector on existing Postgres

Zero extra services. Add the extension, create a vector column, done.

2

text-embedding-3-small

Batch embed on goal create/update. ~$0.00002 per chunk — negligible cost.

3

Claude as the synthesis LLM

Retrieved chunks → grounded answer + chart spec → render inline in chat.

Cost optimisation angle (Parameter 6): Tell evaluators that pgvector eliminates a managed vector DB (~$70/mo saved), batch embedding reduces API calls by 90%, and materialized SQL views serve dashboard queries without hitting the LLM at all. This directly addresses the cost rubric.
